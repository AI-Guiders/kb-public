# Playbook: Zabbix — мониторинг (v1)

> **Scope:** общие практики мониторинга с Zabbix, типовые интеграции (в т.ч. nginx), без привязки к конкретным хостам.

---

## Ключевые понятия

- **Шаблон (template):** набор элементов данных, триггеров и действий, подключаемый к хосту.
- **Элемент данных (item):** метрика, собираемая агентом, SNMP, HTTP и т.д.
- **Триггер (trigger):** условие на основе данных элемента; при срабатывании — инцидент/уведомление.
- **Интеграция с nginx:** через модуль stub_status и Zabbix agent или HTTP-проверки; метрики: соединения (accepted, active, waiting), запросы в секунду.

---

## Типовая проблема: 413 Request Entity Too Large

- **Причина:** размер тела запроса превышает лимит nginx (`client_max_body_size`, по умолчанию 1 МБ).
- **Решение:** в конфигурации nginx (http / server / location) задать `client_max_body_size <значение>;` (например 50M), затем `nginx -t && nginx -s reload`.
- **Мониторинг:** при необходимости — триггеры по коду 413 в access log или по доле ошибок; для загрузки файлов (например карты) лимит задавать в том location, который обслуживает загрузку.

---

## Анти-паттерны

- Не проверять синтаксис nginx перед reload (`nginx -t`).
- Создавать триггеры без порога или с порогом «всегда срабатывает» — шум и усталость от алертов.
- Хранить пароли и токены Zabbix в репозитории.

---

## Диагностика

- **Нет данных по item:** Availability хоста в UI → проверить интерфейс (Agent/SNMP/HTTP), ключ item'а, права на хосте; для HTTP — URL, таймаут, сеть. В форме item — «Test» препроцессинга на примере значения.
- **Очередь:** Monitoring → Queue — задержки по типу (agent, HTTP, SNMP…); при перегрузке — число item'ов, таймауты, производительность БД и housekeeper.
- **Триггер дёргается (flapping):** перейти на окно в выражении (avg/max/min за 5–10 мин вместо last()); при необходимости — зависимость от родительского триггера.
- **Агент недоступен:** проверить сервис на хосте, порт (10050), файрвол, версию агента и сервера.
- Сопоставление алертов с логами приложения и nginx (в т.ч. 413, 5xx).

---

## Создание/правка триггера и шаблона (кратко)

- Триггер: выражение с окном (например `max(/,5m)>80`), описание и вероятные причины, severity; при необходимости — recovery expression (предпочтительно статические значения).
- Шаблон: не передавать пароли через макросы; value mapping для дискретных состояний; зависимые item'ы вместо лишних опросов; при LLD — «Discard unchanged with heartbeat» на мастер-item.
