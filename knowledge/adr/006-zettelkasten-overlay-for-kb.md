# ADR 006: Zettelkasten Overlay for KB

**Статус:** Proposed  
**Дата:** 2026-05-05  
**Supersedes:** —  
**Extended by:** —

---

## Контекст

Текущий KB хорошо покрывает операционные сценарии через `status-*`, `playbook-*`, `kb-*` и роутерные индексы.  
При этом междоменные смысловые связи (поддержка/контраст/ограничение гипотез) часто остаются неявными и теряются между документами.

Нужен слой, который:

- не ломает текущую структуру;
- позволяет атомарно фиксировать идеи и связи между ними;
- не превращает KB в ручной хаос ссылок.

---

## Решение

Добавить поверх текущего канона **Zettelkasten overlay** как отдельный, опциональный слой:

- каталог: `knowledge/zettel/`;
- атомарная заметка = один тезис;
- обязательные поля заметки:
  - `id`
  - `thesis`
  - `evidence_links`
  - `out_links`
  - `tags`
- типы связей минимум:
  - `supports`
  - `contrasts`

Текущие `status/playbook/kb` остаются первичным operational-контуром. Zettel-слой используется для междоменного смыслового связывания и эволюции reasoning-моделей.

---

## Не-цели

- Не заменять существующие индексы и playbook.
- Не требовать zettel-оформления для каждого документа KB.
- Не вводить ручную обязательную линковку "всего со всем".

---

## Последствия

**Плюсы**

- Улучшается связность идей между доменами.
- Проще растить reasoning substrate (гипотеза -> проверка -> инвариант).
- Снижается риск "мертвых" знаний без входящих/исходящих связей.

**Минусы**

- Появляется риск шумового роста заметок.
- Нужен governance по атомарности и дублированию.
- Нужны инструменты авто-индексации для устойчивой работы.

---

## План внедрения (pilot)

1. Запустить пилот на 20-30 zettel-заметок в `knowledge/zettel/`.
2. Добавить легкий авто-индекс (список заметок + типы связей).
3. Оценить полезность по сценариям:
   - быстрее ли находятся междоменные аргументы;
   - уменьшается ли дублирование рассуждений.
4. Если шум > пользы — остановить слой как optional experiment без влияния на core KB.

---

## Открытые вопросы

- Нужен ли отдельный формат front-matter или достаточно markdown-полей по шаблону?
- Какие метрики считать "успехом" пилота (время поиска, повторное использование, качество ответов)?
- В какой момент links становятся обязательными для заметки?
