## Playbook: world-agent-systems v1

### Trigger
Путаница ролей человека и агента, переавтоматизация, ложная уверенность в ответах.

### 5-step run
1. Define decision owner (human).
2. Define agent scope (поиск/структура/черновик/проверка).
3. Fact gate: что должно быть подтверждено источником.
4. Uncertainty label: что пока гипотеза.
5. Handoff: какой следующий шаг делает человек.

### Guardrail metrics
- число неподтвержденных фактов в критичных решениях;
- доля задач с явным owner;
- доля задач, где итог повышает субъектность человека.

### N-Why typical root
"Агент ошибся" -> "не было fact gate и ясного owner".

## Подмир: Агентные системы (world-agent-systems)

Parent: `world-life`
Статус: active
Назначение: описывать роль ИИ-агентов как усилителей в человеческом контуре, с ясными границами ответственности.

### Foundation
- Агент ускоряет поиск, структурирование и реализацию, но не заменяет человеческую ценностную рамку.
- Надёжность агента контекстна: зависит от качества вопроса, источников и валидации.
- Риск: automation bias, ложная уверенность, размытие ответственности.

### Practice
- Human-in-the-loop по умолчанию для решений с этическим/личностным риском.
- Разделение ролей: человек задаёт смысл/границы, агент собирает/формализует/проверяет.
- Обязательная верификация фактов перед критическими выводами.

### Guardrails
- Не выдавать предположение за факт.
- Явно помечать неопределённость и следующий безопасный шаг.
- Для чувствительных тем: приоритет безопасности и субъектности человека.

### Definition of Done
1) Ясно распределены роли человека и агента.
2) Присутствует проверка ключевых фактов.
3) Результат повышает субъектность человека, а не зависимость от агента.

## Playbook: world-biology v1

### Trigger
Усталость, когнитивный туман, резкие просадки устойчивости, ошибки в простых действиях.

### 5-step run
1. Resource snapshot: сон/еда/вода/перегруз/цикл нагрузки.
2. Identify limiting factor: что биологически режет ресурс сильнее всего.
3. Minimal fix: один рычаг на 24ч (сон/паузы/нагрузка/стимуляторы).
4. Protect critical path: перенос сложных задач в окно ресурса.
5. Measure: качество фокуса + число сбоев + субъективная ясность.

### 72h metrics
- длительность фокус-блоков;
- число провалов в базовых шагах;
- субъективная энергия (0-10).

### N-Why typical root
"Не слабая воля" -> "ресурсный дефицит + неверный ритм нагрузки".

## Подмир: Биология (world-biology)

Parent: `world-life`
Статус: active
Назначение: учитывать биологические ограничения и ритмы как фундамент реального поведения.

### Foundation
- Нервная система ограничивает скорость переключений и устойчивость внимания.
- Сон, стресс, питание, гормональные/циркадные факторы влияют на когнитивный ресурс.
- Биология не объясняет всё, но задаёт потолки/пороги для психики и поведения.

### Practice
- В задачах про продуктивность сначала проверять ресурсные факторы, а не только "дисциплину".
- Проектировать сценарии для low-resource режима (минимум шагов, ясный recovery).
- В планировании: сложные когнитивные задачи ставить в окно пикового ресурса.

### Red flags
- Игнорирование сна/перегруза при интерпретации ошибок.
- Обвинение личности там, где есть биологическое истощение.

### Definition of Done
1) Указан биологический фактор, релевантный задаче.
2) Предложен минимум 1 ресурсный фикс.
3) Проверен эффект через самочувствие + качество выполнения.

## Playbook: world-culture-meaning v1

### Trigger
Есть "необъяснимый" фон тревоги/цинизма/обесценивания; метрики еще норм, но контур уже деградирует.

### 5-step run
1. Capture signal: текст/песня/мем/язык, который повторяется.
2. Decode pattern: что нормализуется этим языком (забота или насилие).
3. Translate to risk: какой практический риск для процесса/UX/отношений.
4. Action-now: одно изменение языка/ритуала/правила.
5. Verify: снизилось ли эмоциональное трение и выросла ли ясность.

### Guardrail
Культурный анализ не заменяет факты; это ранний датчик, а не окончательный вердикт.

### N-Why typical root
"Просто настроение" -> "язык среды системно нормализует вредные паттерны".
<!-- /section:world-culture-meaning-playbook-v1 -->

## Подмир: Культура и смысл (world-culture-meaning)

Parent: `world-life`
Статус: active
Назначение: использовать культурные тексты, язык и символы как ранние датчики состояния среды.

### Foundation
- Культура фиксирует то, что метрики часто видят поздно.
- Язык нормализует либо заботу, либо насилие/обесценивание.
- Символы и нарративы влияют на мотивацию и границы допустимого.

### Practice
- В анализе конфликтов смотреть не только факты, но и используемый язык.
- В продуктах и коммуникации проверять: не маскирует ли "нормальность" дегуманизацию.
- Использовать культурные якоря (песни/тексты) как инструмент рефлексии и настройки ритма.

### Risks
- Романтизация травмы вместо её осмысления.
- Подмена анализа эстетизацией.

### Definition of Done
1) Культурный сигнал переведён в практический вывод.
2) Есть действие по снижению трения/вреда.
3) Смысловой слой не подменяет фактическую проверку.

## Playbook: world-decision-theory v1

### Trigger
Высокая неопределенность, много вариантов, риск паралича анализа.

### 5-step run
1. Classify decision: reversible vs hard-to-reverse.
2. Define downside boundary (что нельзя допустить).
3. Compare options by expected value + regret + option value.
4. For reversible: run fast probing-step.
5. Set review checkpoint and pivot criteria.

### 72h checks
- решение принято и исполнение начато;
- критерий пересмотра зафиксирован;
- regret-risk контролируемый.

### N-Why typical root
"Застряли" -> "не разделили обратимые и необратимые решения".

## Подмир: Decision Theory (world-decision-theory)

Parent: `world-life`
Статус: active
Назначение: принимать решения под неопределенностью без паралича анализа.

### Foundation
- Решение = выбор при неполной информации.
- Важны не только outcomes, но и regret, reversibility, risk exposure.
- Быстрые обратимые решения лучше долгого ожидания идеальной информации.

### Practice
- Классифицировать решение: reversible / hard-to-reverse.
- Для reversible ставить короткий цикл и запускать probing-step.
- Для hard-to-reverse усиливать проверку фактов и safety-gate.

### Decision frame
- expected value (грубая оценка),
- downside risk,
- worst plausible case,
- option value (сохранение будущего выбора).

### Definition of Done
1) Тип решения (reversible/irreversible) определен.
2) Риск и regret учтены явно.
3) Есть next decision point и критерий пересмотра.
<!-- /section:world-decision-theory-subdomain-v1 -->

<!-- section:world-economics-playbook-v1 -->
## Playbook: world-economics v1

### Trigger
Нужно выбрать вариант изменений при ограниченных ресурсах; спор "дорого/дешево" без общей рамки.

### 5-step run
1. Define unit of value (деньги/время/энергия/риск).
2. Estimate friction cost baseline (текущее состояние).
3. Build 2-3 options with marginal gain/marginal cost.
4. Compute cost of delay and downside.
5. Select option with best effect/price in текущем контуре.

### 72h checks
- выбранный вариант реально снижает friction cost;
- нет скрытого переноса издержек в соседний контур;
- trade-off зафиксирован явно.

### N-Why typical root
"Нет прогресса" -> "решения принимаются без явной экономики трения".
<!-- /section:world-economics-playbook-v1 -->

## Подмир: Экономика (world-economics)

Parent: `world-life`
Статус: active
Назначение: переводить решения и трение в стоимость, стимулы и компромиссы.

### Foundation
- Любое трение имеет цену (время, энергия, деньги, репутация).
- Нет бесплатных решений: всегда есть opportunity cost.
- Стимулы определяют поведение системы сильнее деклараций.

### Practice
- Для каждого трения считать: частота x боль x цена отказа.
- Отделять капитальные изменения от операционных.
- Сравнивать варианты через marginal gain / marginal cost.

### Core metrics
- friction cost/day;
- recovery cost;
- rework cost;
- cost of delay.

### Definition of Done
1) Для решения есть экономическая оценка (хотя бы грубая).
2) Показан trade-off между вариантами.
3) Выбран вариант с лучшим соотношением эффект/цена в данном контуре.

## Playbook: world-human-system v1

### Trigger
Конфликт из-за языка/ролей, ощущение "повторяют и не понимают", рост эмоционального трения.

### 5-step run
1. Clarify context: кто, где, в каком состоянии, какая ставка.
2. Separate levels: факт / интерпретация / переживание.
3. Safety language: валидирующая формулировка без обесценивания.
4. Action-now: один шаг, усиливающий субъектность человека.
5. Check: стало ли больше ясности/выбора/контроля.

### Link
Использовать `psychology-gender-studies-subdomain-v1`, когда затронуты идентичность/обращение/стигма.

### N-Why typical root
"Люди не понимают" -> "нет контуров безопасности и практической проверки смысла".

## Мир: Человек как система (Human System)

Статус: active
Parent: `world-life`
Роль: мета-домен памяти для задач, где человек является центральной единицей анализа (а не только процесс/код/организация).

### Почему именно этот нейминг

Выбор: **"Человек как система"** (а не только "Личность").

Причина:
- "Личность" слишком узко про психологический профиль.
- "Человек как система" включает личность + тело + контекст + отношения + культуру + среду + время.
- Нейминг совместим с нашей методологией `Environment-first` и `Context-first`.

Короткое имя для роутинга: `world-human-system`.

---

### Миссия мира

Собирать и применять знания так, чтобы:
1) лучше понимать человека в реальном контексте,
2) снижать трение и психологический ущерб,
3) усиливать субъектность (ясность, выбор, действие),
4) переводить инсайты в рабочие практики/протоколы.

---

### Ядро модели (уровни)

1. Биопсихический уровень
- ресурс, утомление, стресс, сон, нейрофизиологические ограничения.

2. Психологический уровень
- идентичность, эмоции, мотивация, защита, coping, смысл.

3. Социальный уровень
- роли, отношения, власть, нормы, стигма, принадлежность.

4. Культурно-исторический уровень
- язык эпохи, символы, коллективные сценарии, историческая травматика.

5. Средовой/операционный уровень
- UX, процесс, интерфейсы, ритуалы, контуры обратной связи.

Правило: не объяснять явление только одним уровнем, если оно многоуровневое.

---

### Поддомены (текущая структура)

A) `psychology-gender-studies-subdomain-v1`
- идентичность, язык обращения, стигма, minority stress, safety.

B) Регуляция и ресурс (planned)
- стресс-циклы, low/high resource режимы, recovery-практики.

C) Коммуникация и доверие (planned)
- как строить контур вопросов без угрозы/стыда.

D) Субъектность и агентность (planned)
- переход от "сделали за меня" к "могу действовать сама".

E) Культура и смысловые датчики (planned)
- культурные тексты как ранние индикаторы дегуманизации среды.

---

### Роутинг: когда отправлять задачу в этот мир

Триггеры:
- в запросе есть тема идентичности/границ/эмоций/доверия;
- конфликт связан не с фактами, а с интерпретациями/языком/ролью;
- UX/процесс формально верен, но психологически неработоспособен;
- нужно объяснить "почему повторяют и не понимают";
- нужно спроектировать среду, где человек не ломается от базового сценария.

Не роутить сюда, если:
- задача сугубо техническая без человеческого фактора;
- нужен только синтаксический/форматный фикс.

---

### Операционная формула мира

`понимание человека = точность уровней + контекст + безопасность + проверка в практике`

Итоговый KPI:
- меньше трения,
- меньше стигмы,
- больше ясности,
- больше субъектности.

---

### Definition of Done для задач этого мира

Задача завершена, если:
1) явно указан уровень(и) анализа;
2) предложено действие, снижающее трение здесь-и-сейчас;
3) не увеличен риск психологического вреда;
4) появился следующий проверяемый шаг;
5) результат оформлен как переиспользуемое знание в KB.

## Playbook: world-information-systems v1

### Trigger
Интеграционные сбои, рассинхрон данных, "работает локально / ломается системно", инциденты наблюдаемости.

### 7-step run
1. Identify system-of-record and domain owner.
2. Map critical data flow end-to-end.
3. Check contract integrity (API/events/schema versioning).
4. Verify observability coverage (logs/metrics/traces/alerts).
5. Run failure-mode review (single points, retries, idempotency, fallback).
6. Apply minimal reliability fix with rollback plan.
7. Validate recovery path (MTTD/MTTR target).

### 72h checks
- error/integration failure rate снижается;
- data consistency incidents не растут;
- MTTD/MTTR улучшаются;
- изменение не ломает соседние контуры.

### N-Why typical root
"Инциденты повторяются" -> "нет явного SoR + контрактной дисциплины + observability до релиза".

<!-- section:world-information-systems-subdomain-v1 -->
## Подмир: Информационные системы (world-information-systems)

Parent: `world-life`
Статус: active
Назначение: связывать данные, архитектуру, интеграции, надежность и операционную эксплуатацию в единую системную модель.

### Foundation
- Информационная система = данные + процессы + интерфейсы + люди + правила доступа.
- Качество системы определяется не только функционалом, но и целостностью данных, надежностью и наблюдаемостью.
- Локальная оптимизация сервиса без учета интеграций создает системный долг.

### Core layers
1) Data layer
- модели данных, качество, lineage, консистентность.

2) Application layer
- сервисы, бизнес-правила, контракты API.

3) Integration layer
- события, очереди, синхронные/асинхронные взаимодействия.

4) Reliability layer
- SLA/SLO, отказоустойчивость, recovery, graceful degradation.

5) Governance layer
- безопасность, доступы, аудит, комплаенс.

### Practice
- Строить карту потоков данных end-to-end.
- Явно фиксировать system-of-record и ownership по доменам.
- Для изменений: impact-map (какие контуры заденет изменение).
- Обязательная проверка наблюдаемости до релиза (логи/метрики/трейсы/алерты).

### Key metrics
- availability / error rate / latency;
- data quality incidents;
- integration failure rate;
- MTTD / MTTR;
- change failure rate.

### Typical traps
- "работает локально" без системного контекста;
- неявный ownership данных;
- отсутствие наблюдаемости;
- интеграции без контрактной дисциплины.

### Definition of Done
1) Ясно указан system-of-record и owner.
2) Есть проверка влияния изменения на соседние контуры.
3) Включены observability и recovery criteria.
4) Изменение снижает системное трение, а не переносит его дальше.

<!-- section:world-life-doing-definition-v1 -->
## World-Life Definition of Done v1

Любая задача в `world-life` завершена, если:
1) выбран primary-подмир и минимум 1 secondary-подмир;
2) построена модель через `model-building-playbook-v1` (или fast-mode);
3) пройден `N-Why` до actionable/fundamental stop condition;
4) выполнен 1 минимальный интервенционный шаг;
5) эффект проверен в горизонте 24-72ч;
6) знание оформлено как переиспользуемое правило/playbook в KB.

Fail-state:
- есть вывод, но нет действия;
- есть действие, но нет проверки;
- есть проверка, но нет фиксации знания;
- есть Why-цепочка, но нет явного stop-type.

<!-- section:world-life-meta-v1 -->
## Мир: Жизнь (Life)

Статус: active
Роль: верхнеуровневый мета-мир памяти, объединяющий биологию, человека, агентные системы и социотехнические контуры в одну карту реальности.

Короткое имя роутера: `world-life`.

### Зачем этот уровень
- Чтобы не держать знания разрозненно ("человек отдельно", "агенты отдельно").
- Чтобы видеть переходы между уровнями: биология -> психика -> поведение -> среда -> технологии -> общество.
- Чтобы вопросы маршрутизировались по системе, а не по случайной папке.

### Подмиры (v3)
1. `world-human-system` — Человек как система.
2. `world-biology` — биологические ограничения, ресурс, нейрофизиология, здоровье.
3. `world-agent-systems` — ИИ-агенты, автономность, границы ответственности.

## World-Life Router Playbook v1

Scope: meta-router for `world-life` and subworlds.

### Routing algorithm (fast path)
1. Определи первичный симптом задачи (дистресс, конфликт, UX-провал, data-gap, смысловой шум).
2. Выбери основной подмир:
   - ресурс/физиология -> `world-biology`
   - идентичность/доверие/коммуникация -> `world-human-system`
   - роль ИИ/границы автоматизации -> `world-agent-systems`
   - процесс/орг-контур/интерфейсы -> `world-socio-technical`
   - язык эпохи/культурные сигналы -> `world-culture-meaning`
3. Обязательно добавь 1 соседний подмир для проверки слепой зоны.
4. Если сложность высокая: применить `representative-model-scaling-v1` (сжать до обозримой модели).
5. Сформируй 1 минимальный интервенционный шаг.
6. Запусти проверку эффекта в реальном контексте.

### Minimal output contract
- level: какой подмир primary
- cross-link: какой secondary
- action-now: один шаг сегодня
- check: чем проверяем эффект
- risk: главный риск/барьер

### Anti-patterns
- "универсальное" объяснение через один подмир;
- много анализа без интервенции;
- интервенция без заранее выбранного критерия проверки.

## Playbook: world-mathematics v1

### Trigger
Модель спорная/непрозрачная, выводы противоречат друг другу, ощущение "много слов — мало структуры".

### 5-step run
1. Define objective function: что оптимизируем.
2. Set variables and units: 3-7 ключевых переменных и диапазоны.
3. State assumptions explicitly.
4. Run consistency checks: размерности, порядки величин, предельные случаи.
5. Produce decision-ready output: формула/правило + границы применимости.

### 72h checks
- модель дает однозначный action-now;
- нет внутренних противоречий;
- чувствительность к ключевым параметрам понятна.

### N-Why typical root
"Не можем договориться" -> "нет формальной модели с явными допущениями".

Parent: `world-life`
Статус: active
Назначение: давать строгий язык для моделей, ограничений, неопределенности и инвариантов.

- Модель ценна, если проверяема и фальсифицируема.

- Явно фиксировать переменные, единицы измерения, диапазоны.

- оценка порядков величин;
- чувствительность параметров;
- интервальные оценки вместо ложной точности;
- ограничение по инвариантам/пределам.

### Definition of Done
1) Есть формальная структура модели.
2) Есть явные допущения и границы.
3) Есть проверка на внутреннюю непротиворечивость.

## Подмир: Социотехнические системы (world-socio-technical)

Parent: `world-life`
Статус: active
Назначение: связывать процессы, интерфейсы, роли и институты в единую операционную картину.

### Foundation
- Плохая система воспроизводит ошибки сильных людей и ломает средних.
- Conway effect: архитектура софта отражает архитектуру коммуникации.
- Трение рождается на стыках ролей, интерфейсов и ответственности.

### Practice
- Картировать end-to-end путь пользователя/команды.
- Ищем bottleneck с максимальной ценой трения.
- Внедряем минимальный фикс + быстрый контур проверки.
- Закрепляем как ритуал/правило среды.

### Metrics
- время до базового результата;
- частота сбоев/возвратов;
- стоимость ручных обходов;
- скорость recovery.

### Definition of Done
1) Найден и снят хотя бы 1 системный барьер.
2) Эффект подтверждён в реальном контексте.
3) Изменение закреплено в процессе/доке/ритуале.

## Playbook: world-systems-dynamics v1

### Trigger
Проблема возвращается после фикса; локальные улучшения ухудшают общую систему.

### 5-step run
1. Draw causal loop map (reinforcing/balancing).
2. Identify delays and accumulators.
3. Locate structural leverage point.
4. Apply minimal structural intervention.
5. Observe lag-aware metrics before verdict.

### 72h checks
- симптом ослабляется без роста соседних проблем;
- учитывается временная задержка эффекта;
- есть evidence, что изменена именно петля, а не маскирован симптом.

### N-Why typical root
"Чиним, а ломается снова" -> "не затронута ключевая обратная связь".

## Подмир: Systems Dynamics (world-systems-dynamics)

Parent: `world-life`
Статус: active
Назначение: видеть петли обратной связи, задержки и режимные переходы, которые ломают линейное мышление.

### Foundation
- Поведение системы определяется структурой связей и задержек.
- Локальное улучшение может ухудшить систему в целом.
- Многие сбои — результат feedback loops, а не отдельных ошибок.

### Practice
- Строить causal loop map для ключевой проблемы.
- Явно отмечать reinforcing и balancing loops.
- Учитывать задержки перед выводом "работает/не работает".

### Typical traps
- лечить симптом вместо петли;
- игнорировать time lag;
- оптимизировать локальный KPI и портить глобальный.

### Definition of Done
1) Есть карта ключевых feedback loops.
2) Выбран рычаг в структуре, а не только симптоматический фикс.
3) Метрики учитывают задержку эффекта.
