### 1. Значения и «типы» в интуиции разработчика

- **Fact:** у примитивов семь общеупотребимых категорий в обучении: `undefined`, `null`, `boolean`, `number`, `bigint`, `string`, `symbol`; всё остальное — **объекты** (включая массивы, функции, даты).
- **Heuristic:** `typeof null === "object"` — исторический артефакт инструмента `typeof`; для проверок на `null` использовать строгое сравнение с `null`/`undefined` или современные формы вроде **nullish** (`??`, `?.`) осознанно, не как замену всему.
- **Confidence:** high

### 2. Строгое и нестрогое равенство

- **Fact:** `===` сравнивает **без приведения типов** (с оговорками для `NaN` и для `+0`/`-0`); `==` применяет **Abstract Equality Comparison** с таблицей в спецификации (множество особых случаев `null`/`undefined`, строка/число, `ToPrimitive` для объектов).
- **Heuristic:** в новом коде **по умолчанию `===` и явные преобразования** там, где нет отдельной дизайн‑причины для `==` (например соглашение «только null/undefined как отсутствие» через `== null` обсуждаемо командно, но должно быть видимо в ревью).
- **Confidence:** high

### 3. Строки и UTF‑16

- **Fact:** строковый тип — последовательность **16‑битных кодовых единиц**; символы вне BMP кодируются **суррогатной парой**; индексы `str[i]` и `.length` измеряют **единицы**, не графемы Unicode.
- **Heuristic:** для пользовательского текста, где важны «символы в смысле Unicode», использовать итерацию по кодпоинтам (`for...of` над строкой или `Intl.Segmenter` где уместно по платформе).
- **Confidence:** high

### 4. `var`, `let`, `const` и TDZ

- **Fact:** `let`/`const` создают **блочную** область и **Temporal Dead Zone**: обращение по имени до завершения инициализатора — `ReferenceError` (для `const` и `let`; у `var` ведёт себя иначе из-за hoisting со значением `undefined` внутри функции/скрипта).
- **Heuristic:** не полагаться на **hoisting функций** как на читаемый контроль потока; для публичных API порядок объявлений = порядок чтения человеком.
- **Confidence:** high

### 5. Замыкания

- **Fact:** функция, созданная в лексическом окружении, удерживает **ссылку** на записи окружения внешних блоков; это объясняет доступ к переменным «снаружи» и утечки при длительных таймерах, если захваченный объект большой.
- **Heuristic:** в циклах с асинхронщиной сознательно фиксировать **копию** итерационной переменной (блочный `const` внутри итерации) вместо ожидания, что все колбэки увидят финальное значение `var`.
- **Confidence:** high

### 6. Обычные функции и стрелки, `this`

- **Fact:** для обычной функции значение `this` задаётся режимом вызова (ссылка с базой, прямой вызов в non‑strict со значением по умолчанию, `call`/`apply`, конструктор `new` и т.д. по правилам спецификации).
- **Fact:** **стрелочная функция** не имеет собственного `this`/`super`/`arguments` как у `function` — лексически берёт из окружающего контекста.
- **Heuristic:** методы объекта, которые должны жить как колбэки без `.bind`, часто оформляются как стрелки‑поля класса **или** явный `.bind` в конструкторе — выбор стиля закрепить в кодстайле команды.
- **Uncertainty:** edge cases с `Proxy` и обёртками — сверять по минимальному примеру под целевым движком.
- **Confidence:** high для базового различия; medium для proxy‑краёв

### 7. Строгий режим

- **Fact:** `"use strict"` включает **строгую семантику** целого скрипта или тела функции (ошибки там, где раньше молчаливые присваивания, иная трактовка некоторых вызовов без базы и т.д. по спецификации).
- **Heuristic:** ESM‑модули **всегда** strict в соответствующем контексте модульного грамматического целевого символа — не смешивать психологически со «свободным сценарием» старого скрипта.
- **Confidence:** high
