## Core Unit

- **context:** переход с классического класс-ориентированного языка на JS.
- **signal:** «почему наследование странное», «spread не копирует геттеры как ожидалось», «`#field` is not defined».
- **action:** мыслить через **объекты как наборы свойств** и **одну цепочку прототипов на объект**.
- **outcome:** корректное использование `extends`, `super`, символов и генераторов там, где нужно итерирование.
- **lesson:** **`class` создаёт функцию‑конструктор и связанный `prototype`**, а приватные поля — **жёстко инкапсулированные ключи** спецификационной модели, не строковые свойства.

### 1. Обычные объекты и доступ к свойствам

- **Fact:** доступ `obj[key]` приводит ключ через `ToPropertyKey`; отличие **точечной** записи (идентификатор) от **вычисляемой** важно для `Symbol` и динамических ключей.
- **Heuristic:** для «словаря» с произвольными строковыми ключами помнить о прототипном загрязнении у `{}` (**`Object.create(null)`** для чистых карт простых сценариев).
- **Confidence:** high

### 2. Дескрипторы свойств

- **Fact:** свойство может быть **data** или **accessor** (`get`/`set`); переопределения через `Object.defineProperty` конфликтуют с уже заданными флагами `writable/configurable/enumerable` по правилам `DefineOwnProperty`.
- **Heuristic:** когда библиотека «замораживает» объекты, проверять, не завязан ли код на возможность патча дескрипторов тест‑дубликатами.
- **Confidence:** high

### 3. Цепочка `[[Prototype]]` и `Object.create`

- **Fact:** поиск свойства идёт вверх по цепочке до `null`; присвоение создаёт **own** свойство на целевом объекте, если не затронут прототип с сеттером (модель в спецификации для OrdinarySetWithOwnDescriptor и т.п.).
- **Heuristic:** `Object.create(proto)` как способ делегирования без конструкторов там, где уместно **чистое** прототипное делегирование.
- **Confidence:** high

### 4. `class`, `extends`, `super`

- **Fact:** `class` задаёт режим **`strict` функции‑конструктора** семантики; `extends` устанавливает связку `[[Prototype]]` между `prototype` производного и базовым, с правилами для `Reflect.construct`/встроенных встроенных объектов (частный случай `new.target`).
- **Heuristic:** нельзя использовать `super` везде одинаково в стрелках‑полях до присвоения базового конструктора — следовать правилам инициализации полей в спецификации классов.
- **Confidence:** medium (edges на встроенных типах типа `HTMLElement` платформенно‑специфичны)

### 5. Приватные поля `#`

- **Fact:** приватные идентификаторы **лексически ограничены телом класса**; они не свойства строковых ключей и не наследуются подклассами как «второй слой» без отдельного объявления.
- **Heuristic:** не путать с соглашением `_field` в TypeScript до стирации типов — это другой контракт командной дисциплины.
- **Confidence:** high

### 6. Итераторы и генераторы (обзор)

- **Fact:** протокол **итератора** возвращает `{ value, done }`; объект **Iterable** содержит `Symbol.iterator`; **генераторы** упрощают реализацию итераторов и совместимы с `async` генераторами там, где разрешены целевой рантайм и синтаксис.
- **Heuristic:** не распространять **однократное** итерирование на объекты, которые пересоздают итератор каждый раз (например многие встроенные коллекции) vs одноразовые потребители (некоторые стримы вручную).
- **Confidence:** medium

### 7. `instanceof` и `Symbol.hasInstance`

- **Fact:** `instanceof` запрашивает `HasInstance` у конструктора (с нормальным алгоритмом для обычных функций).
- **Heuristic:** при кросс‑realm (`iframe`) конструкторы различаются — `instanceof` может врать относительно интуиции «тот же класс»; иногда надёжнее бренд‑проверка протокола (duck typing) или `Symbol.hasInstance` под контролем дизайнера API.
- **Confidence:** medium
