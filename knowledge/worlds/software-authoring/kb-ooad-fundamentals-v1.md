# KB: объектно-ориентированный анализ и проектирование (OOA&D) v1

**Назначение:** фундаментальный канон по **OOA&D** и смежным практикам — откуда берётся правило «существительные → сущности, глаголы → методы», как не сводить его к косметическому рефакторингу.  
**Операционный сжатый проход для агента:** `playbook-ooad-agent-operational-v1.md`.  
**Короткий стоп-кран в коде:** `playbook-domain-nouns-verbs-decomposition-v1.md`.

**Версия:** v1.0 · 2026-05-17 — первый фундаментальный проход (канон agent-notes KB).

**Проверено:** концепции стабильны (Larman GRASP, классический noun/verb analysis, UML 2); конкретные издания — сверять с локальной библиотекой (`programming-books-catalog.md`).

---

## 1. Термины и границы

| Термин | Смысл |
|--------|--------|
| **OOA** (Object-Oriented **Analysis**) | Понять **предметную область**: понятия, правила, сценарии. Модель **концептуальная** — ещё без привязки к C#/файлам. |
| **OOD** (Object-Oriented **Design**) | Спроектировать **программную** структуру: типы, ответственности, связи, слои — реализуемо в коде. |
| **OOA&D** | Связка: анализ → проектирование → реализация; итеративно, не обязательно «всё нарисовать, потом код». |
| **OOP** | Реализация: классы, интерфейсы, полиморфизм, инкапсуляция. |
| **Доменная модель** | Согласованное множество понятий предметной области и их связей (не «таблицы БД» и не «экраны»). |

**Не путать:**

- **GoF-паттерны** — уровень **дизайна** (решения для повторяющихся микропроблем), после того как есть кандидаты в типы.
- **DDD** (Evans) — **стратегический** и **тактический** дизайн сложного домена (bounded context, aggregate, …); опирается на OOA&D, но шире.
- **Clean Architecture / слои** — организация зависимостей; применяется **после** того, как известны основные сущности и сценарии.
- **Рефакторинг** (Fowler) — улучшение **существующего** кода; **задний ход OOA&D** (как chat Skia: домен уже был, presentation — нет).

## 2. Зачем фундаментальный проход (а не только «быстрый патч»)

- **Существительные и глаголы** — не «лайфхак агента», а **классическая техника OOA** (noun/verb analysis → кандидаты в классы и операции).
- Патч в `switch (kind)` закрывает симптом; без анализа **границы понятий** размываются (`Accent`, `ItemKind`, общий record).
- Для UI/renderer тот же проход: **presentation-сущности** (`SkiaChatMessage`, `TopicCard`) — не дубликат домена, а **отображение** доменных понятий (`ChatSurfaceEntry`, …) с собственными `Measure`/`Draw`.

**Эвристика агента:** если в задаче звучат **имена из предметной области** — сначала OOA&D-проход (хотя бы 5–10 минут списков), потом код.

## 3. Техники анализа (OOA)

### 3.1. Сбор словаря предметной области

- Прочитать запрос пользователя, ADR, snapshot/DTO, UI-метки.
- Выписать **существительные** (устойчивые понятия): *сообщение, тема, уточнение, тред, карточка, сессия, …*
- Выписать **глаголы** (действия): *отправить, открыть, измерить, нарисовать, подтвердить, переключить overview, …*
- Отметить **правила** (не всегда отдельный класс): *«активная тема одна»*, *«main thread помечен ◆»*.

### 3.2. Noun phrase analysis (анализ именных групп)

| Шаг | Действие |
|-----|----------|
| 1 | Каждое **существенное** существительное → **кандидат в класс/тип** (entity, value object, иногда enum). |
| 2 | Исключить **технический шум** (`string`, `int`, `Index`, `Kind`) — не домен. |
| 3 | Синонимы объединить (*ветка* / *тред* / *thread* → один тип). |
| 4 | **Атрибуты** vs **типы**: если понятие имеет собственное поведение или жизненный цикл — отдельный тип; иначе поле (`IsPending` у `Confirmation`, не глобальный флаг). |

### 3.3. Verb analysis и сценарии

| Шаг | Действие |
|-----|----------|
| 1 | Глагол → **операция** (метод) или **use case** (сценарий из нескольких шагов). |
| 2 | Спросить: **какой объект** естественный субъект глагола? (*Message.send*, *Topic.open*, *Confirmation.resolve*). |
| 3 | Если глагол не ложится ни на один объект — **сервис** / **controller** / **оркестратор** (GRASP Controller) — осознанно, не «static Util». |

### 3.4. CRC-карточки (Class–Responsibility–Collaboration)

На карточке (или строке таблицы) для каждого кандидата:

- **Class** — имя типа.
- **Responsibilities** — 1–3 обязанности (*хранить текст*, *знать pending*).
- **Collaborators** — с кем взаимодействует (*Thread*, *Session*).

**Правило:** одна карточка — **узкая** ответственность; «God class» виден, когда responsibilities > ~5 без связной темы.

### 3.5. Use case / сценарий (кратко)

- **Актор** + **цель** + **основной поток** (шаги).
- Каждый шаг — сообщение между объектами (задел под sequence diagram).
- Для агента: достаточно **нумерованного списка** из 5–12 шагов перед рефакторингом модуля.

## 4. Проектирование (OOD)

### 4.1. От кандидатов к типам

- **Инкапсуляция:** данные + операции, которые на них опираются — вместе.
- **Полиморфизм:** разное поведение (`Message` vs `Confirmation` vs `TopicCard`) → **разные типы** с общим интерфейсом (`ISkiaChatEntity`), не `switch`.
- **Связи:** ассоциация (ссылка), агрегация (*тред содержит сообщения*), композиция (жизненный цикл части за целым).

### 4.2. GRASP (назначение ответственностей)

Краткая шпаргалка (по Larman / «Применение UML и шаблонов проектирования»):

| Принцип | Вопрос | Типичное применение |
|---------|--------|---------------------|
| **Information Expert** | Кто знает данные для выполнения задачи? | Логика расчёта layout у сущности, знающей свои строки. |
| **Creator** | Кто создаёт B, если B агрегирует/записывает A? | `SceneBuilder` создаёт `Message` из `ChatSurfaceEntry`. |
| **Controller** | Кто обрабатывает системное событие (UI, HTTP)? | `SkiaChatSurfaceControl` — scroll/hit; не рисует каждый вид. |
| **Low Coupling** | Как уменьшить зависимости? | Control не знает деталей `TopicCard.Draw`. |
| **High Cohesion** | Всё ли в классе про одно? | `SkiaChatBubbleRenderer` — общий bubble, не всё подряд. |
| **Polymorphism** | Зависимость от вариантов через типы? | `ISkiaChatEntity.Draw` вместо `ItemKind`. |
| **Pure Fabrication** | Нужен тип, которого нет в домене? | `SkiaChatLayoutEngine`, `SceneBuilder` — допустимы, если именованы. |
| **Indirection** | Посредник для развязки? | Builder между snapshot и entities. |
| **Protected Variations** | Что будет меняться? | Новый вид карточки → новый тип, не новый `case`. |

### 4.3. Слои (после типов)

Типичная схема для приложения / IDE:

1. **Domain / model** — `ChatSurfaceEntry`, правила compositor.
2. **Application** — orchestration, ViewModel-команды (без Skia).
3. **Presentation** — `SkiaChatMessage`, layout, draw.
4. **Infrastructure** — persistence, MCP, файлы.

**Правило:** зависимости **внутрь** (к domain); presentation зависит от модели отображения, не наоборот.

| Тема | Отношение к OOA&D |
|------|-------------------|

| **DDD** | Когда домен **большой**: bounded context, aggregate root, ubiquitous language — поверх OOA&D. |
| **Refactoring** | Когда анализ пропущен — **восстановить** типы через extract class / replace conditional with polymorphism. |
| **HCI / UX** | Не подменяет OOA&D; влияет на **presentation-сущности** и сценарии. |

## 6. Типичные ошибки (в т.ч. у агентов)

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Пропуск анализа | `enum Kind` + рост `switch` | Noun/verb списки, CRC, новые типы |
| Anemic domain | Только DTO, вся логика в service | Перенос поведения к Information Expert |
| God class | Control/ViewModel > 500 строк, всё подряд | Controller + entities + builder |
| Duplicate domain in UI | Те же поля, что в snapshot, но без типов | Presentation types mirroring domain names |
| Pattern-first | «Применим Factory» до списка существительных | Сначала кандидаты, паттерн — если боль повторяется |
| Big design up front only | Диаграммы без итерации | Короткий цикл: модель → код → тест → уточнение |

## 7. Источники и порядок чтения

### Канон (рекомендуемый порядок)

1. **Классический OOA&D + GRASP + итерации:** Craig Larman — *Applying UML and Patterns* («Применение UML и шаблонов проектирования») — **главный якорь** для фундаментального прохода.
2. **UML как нотация:** Rumbaugh & Blaha — *UML 2.0. Объектно-ориентированное моделирование* (есть в `programming-books-catalog.md`).
3. **Историческая тройка (опционально):** Booch / Rumbaugh (OMT) / Jacobson (OOSE) — контекст слияния в UML; для агента достаточно п.1–2.
4. **Паттерны (после GRASP):** Gamma et al. — GoF.
5. **Сложный домен:** Evans — DDD; Vernon — Implementing DDD.
6. **Рефакторинг legacy:** Feathers — *Working Effectively with Legacy Code*.
7. **Практика построения:** McConnell — *Code Complete* (Track C в `map-engineering-reading-v1.md`).

### Официальные / веб

- Microsoft Learn — [Object-oriented programming](https://learn.microsoft.com/dotnet/csharp/fundamentals/object-oriented-programming) (OOP в C#, не заменяет OOA).
- Framework Design Guidelines — именование и контракты **публичных** API после проектирования.

## 8. Evidence-записи (для `kb-engineering-evidence-v1.md`)

- **Fact:** Классический OOA использует noun/verb analysis для выделения кандидатов в классы и операции; OOD назначает ответственности (GRASP) и связи перед кодированием.
- **Heuristic:** Не начинать renderer/presenter с дискриминатора `Kind` по видам доменных понятий; каждое устойчивое понятие — тип с операциями или явный подтип полиморфной иерархии.
- **First adoption task:** для новой UI-фичи выписать 10 существительных и 10 глаголов из ADR/кода; сопоставить с типами; только затем править отрисовку.
- **Success criterion:** нет второго `case` по доменному виду в одном renderer; control < ~400 строк оркестрации.
- **Confidence:** high (по инженерной традиции; конкретная фича — medium до проверки в репо).
