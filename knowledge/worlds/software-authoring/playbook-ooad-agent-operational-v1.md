# Playbook: OOA&D — операционный проход для агента v1

**Назначение:** пошаговый **фундаментальный проход** перед реализацией или крупным рефакторингом. Не заменяет чтение Larman, но заставляет **не пропускать анализ**.

**Фундамент:** `kb-ooad-fundamentals-v1.md`  
**Быстрый контракт в коде:** `playbook-domain-nouns-verbs-decomposition-v1.md`

**Версия:** v1.0 · 2026-05-17

---

## Когда обязателен проход

- Новая **подсистема** или экран (чат, граф, wizard, MCP-панель).
- Рост **`switch` / `enum Kind` / общего record** в renderer, ViewModel, handler.
- Задача «сделать визуал», но в домене уже есть **именованные понятия** (Message, Thread, Order, …).
- Пользователь явно просит **OOA&D**, **декомпозицию**, **сущности**.

**Можно сократить** (только nouns-verbs чеклист): мелкий баг в одном типе, однострочный фикс, конфиг.

---

## Проход (7 шагов)

### Шаг 0 — Зафиксировать границу задачи

- Что **в scope** (один модуль, один слой, одна фича)?
- Какой **артефакт** на выходе (типы, ADR, рефакторинг файла X)?

### Шаг 1 — Словарь предметной области (OOA)

Выписать в ответ пользователю или в черновик ADR (таблица):

| Существительные (кандидаты в типы) | Глаголы (кандидаты в операции) |
|-----------------------------------|-------------------------------|
| … | … |

- Источники: запрос, ADR, `*Snapshot`, compositor, UI-строки.
- Пометить **синонимы** и **шум** (не домен).

### Шаг 2 — Уточнить атрибут vs тип

Для каждого существительного:

- Есть **собственное поведение** или жизненный цикл? → **отдельный тип**.
- Только свойство другого понятия? → **поле** (`IsPending` у `Confirmation`).
- Стабильный набор вариантов без разного поведения? → **enum** (редко — единственный дискриминатор на много видов UI).

### Шаг 3 — CRC (мини)

Для 3–10 главных кандидатов — таблица:

| Class | Responsibilities (1–3) | Collaborators |
|-------|------------------------|---------------|
| Message | хранить текст; отображать в ленте | Thread |
| TopicCard | показать summary; hit open thread | Thread, Overview |

### Шаг 4 — Назначение операций (GRASP)

- Каждый глагол привязан к **Information Expert** или **Controller**.
- Варианты поведения → **Polymorphism** (`ISkiaChatEntity`), не `switch`.
- Создание объектов сцены → **Creator** (`SceneBuilder`).

### Шаг 5 — Слои

Явно разделить:

1. Domain / snapshot  
2. Application (ViewModel, команды)  
3. Presentation (Skia, XAML)  
4. Infrastructure  

Зависимости — **к domain**, не «всё в Control».

### Шаг 6 — Реализация

- Сначала **скелет типов** и интерфейсов (`Measure`/`Draw`/`CreateHit` для UI).
- Потом наполнение, стили, цвета.
- Control — **оркестратор** (цикл, scroll, theme).

### Шаг 7 — Проверка (чеклист)

- [ ] Нет второго `case` по **доменному виду** в одном renderer.
- [ ] Имена типов = **существительные** домена (или явные fabrication: `SceneBuilder`).
- [ ] Глаголы — **методы**, не комментарии «// draw topic» внутри 200 строк.
- [ ] Сборка/тесты; для C# — `roslyn_get_diagnostics` по затронутым файлам.

---

## Связка с рефакторингом legacy

Если код уже есть с `ItemKind`:

1. Словарь из **текущих** `case` и полей record → это и есть пропущенные существительные.  
2. Extract Class / Replace Conditional with Polymorphism (Fowler).  
3. Сверить с доменным snapshot — имена должны **совпадать по смыслу**.
