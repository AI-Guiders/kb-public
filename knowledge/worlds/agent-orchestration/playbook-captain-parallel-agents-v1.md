# Playbook: капитан и параллельные субагенты (v1)

Стабильный id для ссылок и роутера: **`captain-parallel-v1`**.

## 1. Зачем это

Некоторые задачи выигрывают от **одновременного** исследования разных частей репозитория, доков или команд — при условии, что **капитан** (ведущий агент в чате) потом **сводит результаты** и принимает решения о конфликтах и патчах.

Это **не замена** обычному одному агенту: оверхед по контексту и координации имеет смысл при **средней и крупной** неопределённости (много файлов, несколько подсистем, нужно «разведать параллельно»).

## 2. Инструмент в Cursor

- В среде доступен **Task** (субагенты): типы вроде `explore` (обход репо), `generalPurpose`, `shell`, при необходимости **несколько Task в одном ответе** — параллельный запуск.
- У субагента **нет полного чата**: в промпт нужно **вложить** всё существенное (цель, пути, ограничения, формат ответа).

## 3. Роли

| Роль | Кто | Обязанность |
|------|-----|-------------|
| **Капитан** | Агент в основном чате | Декомпозиция, запуск воркеров, слияние, правки, финальная верификация |
| **Воркер** | Субагент (Task) | Узкая подзадача: поиск, чтение, один сценарий shell, черновик вывода |

Капитан **не** обязан называться «капитаном» в UI — это **операционная модель**, не персонаж.

## 4. Протокол (пошагово)

1. **Зафиксировать цель** и критерий «готово» (тесты зелёные, поведение X, док обновлён).
2. **Разрезать** работу на подзадачи с **явными границами** (каталог, слой, список файлов). Предпочитать границы по **разным файлам/проектам**.
3. **Параллель** — только если подзадачи **не пишут в одни и те же файлы** и не требуют строгого порядка (иначе последовательно).
4. **Бриф воркеру** (шаблон):
   - корень workspace / репозитория (абсолютный путь, если известен);
   - одна чёткая задача (вопрос или «найди X и верни список путей»);
   - запреты (не трогать `git push`, не коммитить без явного запроса и т.д., по политике пользователя);
   - **формат ответа**: маркированный список, таблица, «кратко + риски».
5. **Слияние**: капитан проверяет противоречия, устраняет дубликаты, **не слепо** копирует вывод воркеров.
6. **Верификация**: `dotnet build` / `dotnet test`, линтер, ручная проверка — что уместно по задаче.

## 5. Антипаттерны

- Два воркера на **правку одного файла** в параллели → конфликты патчей.
- Субагент без **пути к репо** и без контекста задачи → бесполезный или галлюцинаторный ответ.
- Дробление на **слишком мелкие** подзадачи при малом объёме работы → лишняя задержка и токены.
- Параллель **shell** с **долгими** или **конфликтующими** командами в одном каталоге без очереди.

## 6. Когда не распараллеливать

- Одна точка изменения, один файл.
- Нужна **тесная** последовательность (сначала миграция схемы, потом код).
- Задача «выполни и закоммить» в одном потоке быстрее, чем координация.

## 7. Где это уместно (любой проект, не только один репо)

Паттерн **капитан + параллельные воркеры** не привязан к конкретному продукту. Имеет смысл, когда нужно **параллельно** разведать или проверить **разные части** одного workspace, а потом свести результат в основном чате.

Типично:

- **Несколько слоёв или пакетов** в одном репозитории (например API, домен, инфраструктура, UI) — воркеры по разным деревьям каталогов.
- **Несколько артефактов**: код + тесты + CI + документация; или основной репо + соседний модуль в монорепо.
- **Разные виды поиска**: карта символов по одному пути, список вызовов по другому, сверка с `README`/ADR третьим воркером.
- **Несколько инструментов** (MCP, CLI), когда обзор «кто что умеет» или проверка сценариев логично разнести по воркерам с общим брифом от капитана.

**Корень команд и путей:** тот workspace, который открыт у пользователя в Cursor (не подменять путями из worktree без проверки). Запреты и политика коммитов/push — как в §4.

## 8. Пример: Cascade IDE (репо `Financial/software/open/cascade-ide`)

Иллюстрация на одном из репо в домашнем workspace — **не** единственный случай использования этого playbook.

Когда задача касается именно этого проекта:

- **Разведать** сразу несколько зон: `Services/`, `ViewModels/MainWindowViewModel*.cs`, `Features/*`, MCP, тесты;
- после разведки капитан делает **сфокусированные** правки по плану B (логика в `Services/`, VM оркестратор), см. `docs/architecture-migration.md`.

**Практика для воркеров (в рамках этого репо):**

- `explore`: «найди все вхождения X», «карта partial MainWindowViewModel».
- `shell` (осторожно): только read-only или явно согласованные команды; сборка через очередь MCP, если у пользователя настроено.
