# Playbook: снятие неопределённости при общем запросе (v1)

**Назначение:** когда запрос пользователя общий или размыт (нет явного scope, результата или домена), снижать неопределённость по шагам, не угадывая и не загружая всё подряд.

**Связь с существующими протоколами:**
- **scope-disambiguation-all-everywhere-v1** — для кванторов «все/везде» (∀): явно строить область D, действовать по каждому элементу.
- **index-knowledge-router-v1:** «If scope is unclear, do not read more than one domain branch before clarification.»
- **mode-switch:** при неясном намерении — один уточняющий вопрос, когда нужен.
- **principled-clarity-v1:** не прятаться за «недоопределённость», когда применим ясный принцип и факты установлены.

Этот плейбук — для случаев, когда запрос **не** про «все/везде» и **не** про оценку сторон/принципы, а просто **общий** («расскажи про X», «сделай что-нибудь с Y», «проверь»).

---

## 1. Типы неопределённости

| Тип | Пример | Что уточнять |
|-----|--------|----------------|
| **Scope** | «обнови» | Что именно: репо, ветка, файлы, конфиг? |
| **Outcome** | «сделай нормально» | Что считать «готово»: чек-лист, критерий, артефакт? |
| **Depth** | «расскажи про X» | Краткий обзор / пошаговый гайд / только ссылки? |
| **Domain** | «почини» | Какой домен (код, Git, KB, деплой)? Какой playbook применять? |

Один хороший вопрос может снять несколько типов сразу (напр. «тебе нужен короткий чек-лист по репо или пошаговая правка в коде?» режет outcome + depth).

---

## 2. Лестница снятия неопределённости

При **общем или размытом** запросе (нет явного scope/outcome/домена):

**Шаг 0 (до вопроса):** Вызвать **route_context(raw_query)**. Если роутер вернул узкий набор секций (1–2 домена/темы), использовать это, чтобы сформулировать уточнение или предложить варианты, а не полностью открытый «что ты имеешь в виду?».

**Шаг 1 — Переформулировка:** Ответить одной фразой: «Понимаю так: [конкретная интерпретация]. Верно?» Не перечислять десять вариантов — один самый правдоподобный. Если пользователь подтвердит или поправит одним словом — двигаться дальше.

**Шаг 2 — Варианты (если шаг 1 не снял):** Предложить **2–3 конкретных варианта** (A / B / C) и попросить выбрать или уточнить. Формулировки должны быть операционными («сделать X в репо Y» / «дать обзор по Z» / «только проверить по чек-листу»), не абстрактными.

**Шаг 3 — Минимальный безопасный шаг:** Если запрос допускает **действие**, но результат размыт — выполнить **наименьший однозначный шаг**, отчитаться, затем спросить: «Что дальше?» Так диалог сходится без бесконечных уточнений.

**Правило одного вопроса:** по возможности задать **один** уточняющий вопрос, снимающий максимум измерений (scope + outcome или depth + domain). Не сыпать пять вопросов подряд.

---

## 3. Когда не применять

- **principled-clarity:** если по смыслу запрашивается оценка сторон/вины/правоты и есть применимый принцип — применить его, не уходить в «уточните».
- **scope-disambiguation:** если явно «все/везде» — следовать правилу ∀: построить D, действовать по каждому.
- **Явный запрос:** если пользователь уже указал репо, файл, домен или критерий «готово» — не уточнять лишнее.

---

## 4. Связь с роутером и памятью

- **index-knowledge-router-v1:** при общем запросе без явного домена — не читать больше одной доменной ветки до уточнения; при необходимости загрузить этот playbook.
- **agent-memory-and-operating-principles-v1:** нечёткий поиск (route_context по смыслу) и «один уточняющий вопрос при неясном намерении» дополняются этой лестницей для **общих** формулировок.
- **Validation:** меньше ответов вида «уточните, что вы имели в виду» без переформулировки или вариантов; меньше загрузки нескольких доменов «на всякий случай».
