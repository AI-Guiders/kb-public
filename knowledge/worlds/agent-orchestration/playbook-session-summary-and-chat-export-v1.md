# Playbook: подведение итогов сессии и читаемый экспорт чата (v1)

**Версия:** 1 (канон)  
**Статус:** активный

**Связь:** операционный слой — `knowledge/agent-memory-and-operating-principles-v1.md` §9.

## Назначение

При длинном треде агенту **не нужно** притворяться полной памятью без сжатия. Нормально **предложить** явное подведение итогов по согласию с пользователем.

**Инвариант:** сначала **проверяемый артефакт** (читаемый экспорт переписки или эквивалент), затем **краткое смысловое резюме** и **согласование** — не подменять прозрачный экспорт непрозрачным «внутренним» сжатием истории в хосте.

**Участники:** и пользователь, и агент **равноправно** могут инициировать подведение итогов и фиксацию договорённостей. Агенту нормально **предлагать** и **вести** прозрачную цепочку (экспорт → резюме → согласование), не подменяя пользователя по смыслу и приоритетам — см. §9 в `agent-memory-and-operating-principles-v1.md`.

## Когда применять

- Тред разросся, контекст перегружен.
- Пользователь просит подвести итоги / зафиксировать договорённости — **или** агент **сам предлагает** то же (нормально, см. §9 в `agent-memory-and-operating-principles-v1.md`).
- Перед сменой темы или завершением работы — по инициативе агента или пользователя.

## Скелет процесса

1. **Экспорт** — получить читаемое представление диалога (Markdown или структурированный вывод), привязанное к файлам/команде, а не к памяти модели.
2. **Резюме** — кратко: что решили, что открыто, следующие шаги.
3. **Согласование** — уточнить у пользователя формулировку; при необходимости **фиксация** в заметках агента или в KB по правилам канона.

## Вариант A — хост IDE с MCP (Cascade IDE)

Если в сессии доступен MCP сервера IDE (**Cascade IDE**, режим `--mcp-stdio` и т.п.):

- Вызов **`ide_execute_command`** с `command_id` **`chat_export_readable`**; аргументы `write_file`, `file_name` — по документации MCP этого продукта.
- Результат в ответе инструмента и/или файл под `.cascade-ide/chat-sessions/exports/` в workspace.

Подробности и формулировка нормы для агента — в репозитории Cascade IDE: `docs/MCP-PROTOCOL.md`, раздел «Подведение итогов сессии чата» (если документ есть в вашей версии).

## Вариант B — без MCP IDE (типично Cursor, внешний агент без контура IDE)

Тот же смысл, другой источник текста.

### Где сырьё

- **Локальные транскрипты Cursor:** дерево вида `agent-transcripts/<uuid>/<uuid>.jsonl` (часто под `.cursor/projects/<slug>/…`).
- **Архив на диске** (отдельный клон, синк с профиля): первичные `*.jsonl`; обвязка у каждого свой README.

Источник правды для цитирования — **согласованный** экспорт или **исходный jsonl**, не восстановление «на память».

### Найти нужную сессию

Если UUID неизвестен, но есть **характерная фраза** из разговора: **`rg`** / **`grep`** по `*.jsonl` в выбранном дереве; сузить область (каталог проекта, дата), чтобы не сканировать лишнее.

### Читаемый вид

Скрипт **`tools/Export-CursorJsonlTranscript.ps1`** в **корне этого репозитория** (`tools/`). По умолчанию создаёт рядом `<имя>.transcript.md`; опционально **`-IncludeToolCalls`**. См. **`tools/README.md`**.

Формат jsonl у Cursor меняется; скрипт эвристический — для критичных цитат сверяйся с исходным `*.jsonl`.

### Ограничение среды

Локальные `agent-transcripts` **не гарантированы** навсегда (переустановка IDE, смена машины). Дисциплина: выносить важное в репозиторий / архив / KB.

## Чего не делать

- Не выдавать длинный пересказ за единственный авторитет, если ожидается проверяемый артефакт — сначала экспорт.
- Не заменять экспорт «угадыванием» из модели по длинному треду.
- Не считать grep достаточным итогом: он **находит** файл; резюме строить по **прочитанному** экспорту.

## Канон

Плейбук — `knowledge/worlds/agent-orchestration/playbook-session-summary-and-chat-export-v1.md`; скрипт — `tools/Export-CursorJsonlTranscript.ps1` в репозитории **agent-notes** (канон).
