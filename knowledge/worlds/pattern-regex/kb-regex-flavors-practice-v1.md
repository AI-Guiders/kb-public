# Regex: диалекты и практика (L2)

## Назначение

Сопоставить **одну задачу** с **конкретным API**: опции, флаги, отличия ECMAScript / PCRE / .NET / Java. Снизить риск «скопировал со Stack Overflow — в моём языке не работает».

## Инварианты

- Нет единого «стандарта regex» для всех фич; всегда сверяй с документацией целевого движка.
- Флаги в **источнике паттерна** (`(?i)`) и флаги **API** могут дублироваться или конфликтовать — держи один источник правды.

<!-- section:dotnet -->
## .NET (`System.Text.RegularExpressions`)

- Класс `Regex`; опции через `RegexOptions` (IgnoreCase, Multiline, Singleline, CultureInvariant, ECMAScript и др.).
- **Таймаут:** конструкторы и статические методы с `TimeSpan` — обязательны для пользовательского ввода паттернов.
- **Компиляция:** `RegexOptions.Compiled` — ускорение повторных запусков, цена на загрузку; не панацея при катастрофическом backtracking.
- Именованные группы: `(?<name>...)`; ссылки `\k<name>` в replacement через `${name}` в некоторых перегрузках.

**Совет:** для парсинга форматов с вложенностью рассмотреть `GeneratedRegex`, span-перегрузки (.NET 7+) или специализированный парсер.
<!-- /section:dotnet -->

<!-- section:javascript -->
## JavaScript (ECMAScript RegExp)

- Литерал `/pattern/flags` или конструктор `RegExp("pattern", "flags")` — **двойное** экранирование в строке.
- Флаги: `g`, `i`, `m`, `s` (dotAll), `u` (Unicode), `y` (sticky) — набор зависит от версии стандарта и движка.
- Lookbehind появился в современных движках; ограничения по длине lookbehind мягче, чем у «классического» ICU/PCRE в старых режимах, но проверяй цель.

**Подводный камень:** методы строки и `RegExp` с флагом `g` хранят `lastIndex` — состояние между вызовами.
<!-- /section:javascript -->

<!-- section:pcre-perl-style -->
## PCRE / Perl-стиль

- Богатый набор: условные паттерны, калли, рекурсия, атомарные подвыражения, владение — **не переносить** в минимальные движки.
- Флаги встроены в паттерн `(?i)` и т.д.; модификаторы `(?x)` free-spacing.

Использование в проде: ограничивать подмножество фич, если паттерны пишут пользователи.
<!-- /section:pcre-perl-style -->

<!-- section:java -->
## Java `java.util.regex`

- `Pattern` + `Matcher`; флаги `Pattern.CASE_INSENSITIVE`, `MULTILINE`, `DOTALL`, `UNICODE_CHARACTER_CLASS` и др.
- Поведение `\b` и `\w` сильно зависит от UNICODE-флагов.

**Совет:** для сложных Unicode-сценариев явно включать Unicode-aware опции и тестировать на реальных данных.
<!-- /section:java -->

<!-- section:workflow -->
## Рабочий процесс (проверка паттерна)

1. Зафиксировать **примеры**: должны совпасть / не должны совпасть / граничные.
2. Выписать **опции** (многострочность, Unicode, ignore case).
3. Прогнать на **длинной** синтетике при пользовательских паттернах — ловля катастрофы.
4. Включить **таймаут** или лимит в API, где возможно.
5. Документировать **диалект** в комментарии к коду (одна строка: «PCRE 8.x / .NET / ES2024»).

См. также дисциплину тестов и безопасности: `regex-playbook.md`.
