# Обычная логика и связка с формальной: Evidence-Based (v1)

Evidence-based обзор: как в естественном языке выражаются кванторы, условие, необходимость и достаточность; как это соотносится с формальной логикой (∀, ∃, импликация); типичные сбои рассуждения и их формальные аналоги. Цель — заземлить операционные правила (напр. scope-disambiguation, различение ∀/∃) в явном логическом фундаменте и в примерах из практики.

**Источники:** Stanford Encyclopedia of Philosophy (Logic and Reasoning, Informal Logic, Quantifiers); классические учебники по логике (кванторы, импликация, контрапозиция); литература по критическому мышлению и аргументации (necessity vs sufficiency, scope errors); собственные кейсы из диалогов и задач (см. раздел Evidence).

## 1. Обычная логика: кванторы и область

### 1.1 Универсальный охват (∀) в речи

- **«Все», «каждый», «любой», «везде», «всё», «every», «all», «everywhere»** — в естественном языке выражают утверждение или требование, относящееся к **каждому** элементу некоторой области. Область часто не названа явно.
- **Структура важнее слов:** как только по смыслу высказывание имеет вид «для каждого X из D выполняется P(X)», это универсальная квантификация. Чтобы не сужать D по умолчанию до одного элемента, нужно **явно построить D** (перечислить: какие репо, какие артефакты, какие шаги).
- **Типичный сбой:** подставить один привычный элемент («скрипт», «одно действие») и остановиться, не проверив остальные элементы D. После действия — явно пройти по каждому элементу D и подтвердить: «для этого элемента сделано».

### 1.2 Экзистенциальный охват (∃) в речи

- **«Хотя бы один», «какой-то», «есть», «достаточно одного», «some», «at least one», «there exists»** — утверждение или требование о **существовании** элемента области с некоторым свойством. Не «для всех», а «есть такой».
- **Не путать с ∀:** из «сделай в одном месте» не выводить «сделай везде»; из «достаточно одного примера» не выводить «нужны все». И наоборот: из «сделай везде» не ограничиваться одним пунктом.

### 1.3 Подмена квантора

- **∃ → ∀:** один пример/один успех трактуется как «для всех так» (generalization from one case).
- **∀ → ∃:** из «для всех требуется X» делают «достаточно сделать X в одном месте» или «раз один раз не сработало — ничего не работает».
- **Операционное правило:** явно различать квантор в **запросе** и квантор в **выводе**. В запросе «пуш везде» — ∀ по множеству целей; в выводе «готово» — только если по каждой цели действие выполнено.

## 2. Условие, необходимость и достаточность

### 2.1 «Если A, то B» в речи и в формальной логике

- **Импликация A → B:** в формальной логике «если A, то B» не требует «A причиняет B»; это отношение истинностных значений. Важно: **ложная посылка не опровергает правило.** «Если идёт дождь, земля мокрая» — из того, что дождя не было, нельзя заключить, что земля не мокрая (она могла быть полита).
- **Контрапозиция:** A → B эквивалентно ¬B → ¬A. «Если не B, то не A» — часто полезная переформулировка для проверки.

### 2.2 Необходимость и достаточность

- **A необходимо для B:** B → A (если B, то A). Без A не бывает B. «Наличие пароля необходимо для входа»: вход → пароль есть.
- **A достаточно для B:** A → B (если A, то B). При A получаем B. «Пароль достаточен для входа» (если других условий нет): ввёл пароль → вошёл.
- **Типичная путаница:** «A необходимо» путают с «A достаточно»; «одного примера достаточно» путают с «для всех нужен пример» (∃ vs ∀).

### 2.3 Типичные ошибки в рассуждениях

- **Утверждение консеквента (affirming the consequent):** из «если A, то B» и «B» заключить «A». Неверно: B может быть по другой причине.
- **Отрицание антецедента (denying the antecedent):** из «если A, то B» и «не A» заключить «не B». Неверно: B может быть и без A.
- **Один сбой → «ничего не работает»:** из «в одном случае X не сработало» заключить «X никогда не работает». Подмена ∃ (есть случай сбоя) на ∀ (для всех не работает).

## 3. Связка: обычная формулировка ↔ формальная структура

| Речь | Формально | Действие агента |
|------|-----------|-----------------|
| «Везде», «всё», «все» (по области целей) | ∀ x ∈ D : P(x) | Явно задать D, для каждого x проверить/сделать P(x). |
| «Хотя бы один», «достаточно одного» | ∃ x ∈ D : P(x) | Не требовать P для всех; не выводить из одного примера ∀. |
| «Если сделал X, готово» | X → «готово» | Не путать с «готово только если для всех элементов D сделано X». |
| «Один раз не сработало» | ∃ случай ¬P | Не выводить ∀x ¬P («нигде не работает»). |

## 4. Evidence: кейсы из практики

### 4.1 «Пуш везде на VDS»

- **Формулировка пользователя:** «давай пуш везде на VDS».
- **Сбой:** интерпретировано как один шаг (скрипт сборки и пуша kb-public). Канон (agent-notes) не был запушен — скрипт его не пушит. Область D = {канон, kb-public}; для канона действие (git push) не выполнено.
- **Формально:** запрос имел вид ∀ d ∈ D : push(d). D не было перечислено явно до действия; после действия не было проверки по каждому элементу D.
- **Исправление:** правило scope-disambiguation-all-everywhere-v1 — до действия перечислить D, после действия пройти по каждому элементу и подтвердить. В процедуру добавлено: канон пушится отдельно; в мышлении — «для каждого элемента D я сделал шаг?».

### 4.2 Вывод «для всех» из одного примера

- **Типичный сбой:** пользователь сообщает один пример («вот тут не сработало»); агент или человек заключает «значит, везде/всегда не работает». Это подмена ∃ (есть контрпример) на ∀ (для всех не выполняется).
- **Формально:** из ∃x ¬P(x) не следует ∀x ¬P(x). Требуется либо больше данных, либо явная оговорка «в одном наблюдённом случае».

### 4.3 «Достаточно одного» vs «нужно для всех»

- **Путаница:** «достаточно сделать в одном месте» (∃) трактуется как «надо сделать везде» (∀), или наоборот. Операционно: при запросе с «все/везде» не ограничиваться одним пунктом; при запросе «хотя бы один» не требовать охвата всей области.

## 6. Операционный вывод для агента

- При формулировках с универсальным охватом (все, везде, всё): явно задать область D, для каждого элемента D выполнить или проверить действие; после шагов — пройти по D и подтвердить по каждому элементу. Не считать готовым по одному привычному шагу.
- Не подменять кванторы: из ∃ не выводить ∀; из ∀ не опускаться до ∃ без явного основания.
- При условии «если A, то B»: не утверждать консеквент (из B не выводить A), не отрицать антецедент (из ¬A не выводить ¬B); при одном сбое не заключать «ничего не работает» без дополнительных данных или оговорки.
