# Avalonia UI + Dock — fundamentals v1

**Назначение:** общий фундамент для реализации **дизайн-задач** в desktop .NET на **Avalonia** с **Dock** (как в CascadeIDE): от модели UI до типичных ошибок. Операционные чек-листы — в `playbook-avalonia-dock-ui-v1.md`. Продуктовый слой (иерархия действий, доступность) — `../hci-ux-dx/playbook-hci-core-v1.md`, `../hci-ux-dx/ui-ux-playbook.md`, `../hci-ux-dx/kb-hci-usability-and-dialog-rules-v1.md`.

**Мир:** `software.desktop-ui` / engineering.

---

### 1. Роль Avalonia в стеке

- **Fact:** Avalonia — кроссплатформенный UI-фреймворк для .NET: визуальное дерево, стили, биндинги, рендеринг Skia; разметка AXAML (как XAML с отличиями).
- **Heuristic:** воспринимать приложение как **дерево контролов + стили + data context**; любая «магия» должна быть явно в VM или в attached behavior, а не в разрозненном code-behind.
- **First adoption task:** для экрана указать: корневой `DataContext`, источник стилей (`App.axaml`, `Themes/`), и где живут `DynamicResource` ключи.
- **Success criterion:** новый разработчик находит тему, модель видимости и границу VM без чтения всего решения.
- **Confidence:** high

---

### 2. MVVM и границы слоёв

- **Fact:** типичный стек — `CommunityToolkit.Mvvm` (`ObservableObject`, `[RelayCommand]`, partial methods) + AXAML биндинги; `x:DataType` включает compiled bindings (в CascadeIDE `AvaloniaUseCompiledBindingsByDefault=true`).
- **Heuristic:** команды и видимость — в VM; разметка не содержит бизнес-правил; для списков — стабильные типы элементов (`DataTemplate` с `DataType`).
- **First adoption task:** для новой панели завести VM или под-VM и проверить биндинги в Debug (ошибки биндинга в логе Avalonia).
- **Success criterion:** смена состояния не требует правок XAML кроме объявления привязок.
- **Confidence:** high

---

### 3. Ресурсы, темы и `DynamicResource`

- **Fact:** глобальные стили подключаются через `StyleInclude` в `App.axaml`; кастомные палитры могут жить в **JSON theme dictionaries** (как в CascadeIDE: `Themes/dark-theme.json`, `power-theme.json` и т.д.) и маппиться на ключи вроде `CascadeTheme.*`.
- **Heuristic:** для цветов/кистей, зависящих от темы, предпочитать **`DynamicResource`**; для констант layout — `StaticResource` или явные значения; не дублировать один и тот же цвет в десяти местах — вынести в ресурс.
- **First adoption task:** при добавлении нового визуального блока определить: ключ ресурса уже есть или нужен новый в теме.
- **Success criterion:** переключение темы не ломает контраст и не оставляет «жёстко зашитых» цветов в новом коде.
- **Confidence:** high

---

### 4. Стили, классы и приоритет

- **Fact:** Avalonia использует селекторы `Style` (`Selector="Border.modeCard"`, псевдоклассы `.pointerover` и т.д.); несколько классов на элементе (`Classes="modeCard chatPanelSection"`) комбинируются.
- **Heuristic:** при конфликте внешнего вида проверять **порядок и специфичность** стилей; локальный `Setter` на элементе бьёт глобальный стиль; дублирующиеся `Classes` с разным смыслом разводить по файлам/регионам темы.
- **First adoption task:** при «не применяется стиль» — поиск по селектору и по `Theme` вкладке / runtime Live.
- **Success criterion:** предсказуемое наследование: один блок — один основной класс «карточки» + опциональные модификаторы.
- **Confidence:** medium

---

### 5. Компоновка: Grid, StackPanel, DockPanel, ScrollViewer

- **Fact:** `Grid` — основной инструмент для колонок/строк; `StackPanel` — линейная укладка; встроенный **`DockPanel`** — приклейка дочерних к краям (не путать с библиотекой Dock); `ScrollViewer` меняет измерение дочерних элементов (`MaxHeight` + scroll).
- **Heuristic:** для **равных колонок** в карточках — `UniformGrid` или `Grid` с `*`; избегать бесконечной высоты внутри scroll без `MaxHeight`/ограничений.
- **First adoption task:** для новой секции нарисовать схему: что растягивается, что фиксировано, где scroll.
- **Success criterion:** нет обрезания без scroll там, где контент динамический; нет «прыгающей» высоты окна из-за неограниченного контента.
- **Confidence:** high

---

### 6. Dock (библиотека) vs `DockPanel` (встроенный)

- **Fact:** **Dock.Avalonia** + **Dock.Model** дают модель **корневого дока**, **документов**, **инструментов** (`IDock`, `IDockable`, `IFactory`); в разметке — `DockControl` и тема `Dock.Avalonia.Themes.Fluent`. Встроенный **`DockPanel`** — только локальная укладка детей.
- **Heuristic:** если речь о «вкладках редактора, плавающих панелях, сохранении layout» — это **Dock.Model**; если «заголовок сверху, кнопки снизу» — часто достаточно `DockPanel`/`Grid`.
- **First adoption task:** при баге «док не обновляется» проверить: фабрика, `VisibleDockables`, активный документ, биндинг `DockControl.Layout`.
- **Success criterion:** изменение набора документов отражается в UI без ручного пересоздания всего окна.
- **Confidence:** high (для CascadeIDE); medium в абстрактных проектах

---

### 7. Всплывающие слои: Flyout, Popup, окна

- **Fact:** для вторичного контента без раздувания основного дерева используются `Flyout`, `Popup` или отдельные `Window`; в версии 11.x API разметки следует сверять с документацией (placement, light dismiss).
- **Heuristic:** **прогрессивное раскрытие** (HCI) → flyout/popover для деталей; не вкладывать тяжёлые списки в всегда видимую колонку без необходимости.
- **First adoption task:** после добавления flyout — сборка Release + клик по триггеру на целевой ОС.
- **Success criterion:** фокус, закрытие и биндинги внутри flyout работают предсказуемо.
- **Confidence:** medium (API эволюционирует)

---

### 8. Списки и виртуализация

- **Fact:** `ItemsControl`/`ListBox` + `DataTemplate`; для больших списков важна виртуализация (`VirtualizingStackPanel` и настройки контейнера — по версии и паттерну).
- **Heuristic:** тяжёлые шаблоны × много элементов → измерять лаг скролла; не вкладывать в item ненужные поддеревья.
- **First adoption task:** при > сотен элементов — проверить, включена ли виртуализация и нет ли лишних `ScrollViewer` внутри item.
- **Success criterion:** скролл остаётся плавным на типичном железе.
- **Confidence:** medium

---

### 9. AvaloniaEdit, TextMate и редакторский слой

- **Fact:** **AvaloniaEdit** — отдельный пакет (версия часто **отстаёт** от core Avalonia); **AvaloniaEdit.TextMate** даёт подсветку через **TextMate-грамматики** (bundle: `.tmLanguage` / `.tmLanguage.json` / plist и связанные ресурсы), а не через автоматическую привязку Roslyn для произвольных языков.
- **Fact:** **где лежат грамматики и как они регистрируются** (пути, `TextMateInstallation`, копирование в выходной каталог, порядок инициализации) — **источник правды в коде репозитория** при старте приложения / создании редактора; KB фиксирует только класс рисков.
- **Heuristic:** симптом «**нет подсветки**» → проверить: bundle на диске/в ресурсах, корректное **включение файлов в сборку** (`Content`/`EmbeddedResource`/копирование), совпадение расширения файла с тем, что ожидает регистрация; лог старта редактора на исключения из слоя TextMate.
- **Heuristic:** симптом «**сломалась после апгрейда**» → сначала матрица **Avalonia.AvaloniaEdit ↔ AvaloniaEdit.TextMate** (и при необходимости release notes), затем уже правки грамматик.
- **Heuristic:** **новый язык** — минимальный инкремент: проверенный grammar bundle + один smoke-файл; не смешивать с крупным редизайном окна без отдельного прогона редактора.
- **Heuristic:** кастомные **margin / glyph / line number** зоны — высокорисковая зона регрессий (измерение, hit-testing, тема).
- **First adoption task:** найти в решении место **установки TextMate / выбора grammar** для `TextEditor` и от него трассировать путь к файлам bundle.
- **First adoption task:** при обновлении пакетов редактора — smoke: открытие файла, поиск, перенос строки, смена темы приложения, быстрый ввод в большом файле.
- **Success criterion:** редактор не ломает фокус остального окна, не теряет привязку документа к VM, подсветка стабильна без повторяющихся ошибок в логе.
- **DX (специфика Avalonia):** полного паритета с классическим **WPF XAML designer** нет; базовый цикл — **AXAML → `dotnet build` → run**; часть проблем (**биндинги**, **DynamicResource**, lifecycle) проявляется только в **runtime**. Для инспекции дерева в dev-сборках используют **DevTools** (включение и жест — по [документации Avalonia](https://docs.avaloniaui.net/) для твоей мажорной версии, не кэшировать в KB конкретную клавишу).
- **DX (общий поток):** C# и рефакторинг — `dotnet-roslyn-debug-playbook.md`; сборка, блокировки выходных файлов, тесты — `../../tooling-debug-playbook.md`; снижение трения и привычки доставки — `../hci-ux-dx/de-dx-playbook.md`.
- **Confidence:** medium (детали API и путей к bundle — в репо; TextMate — повышенный риск версионного дрейфа)

### 10. Markdown и прочие контролы

- **Fact:** **Markdown.Avalonia** (и аналоги) рендерят HTML-подобный контент внутри Avalonia; стили и шрифты наследуются от темы с оговорками.
- **Heuristic:** для «читаемого текста» задавать базовые стили в теме; большие документы — с scroll и лимитами высоты.
- **Confidence:** medium

---

### 11. Сборка, дизайн-тайм и проверка

- **Fact:** ошибки AXAML ловятся на этапе **build**; часть проблем — только в runtime (биндинги, lifecycle).
- **Heuristic:** после существенной разметки — `dotnet build -c Release`; если exe запущен и блокирует выходной файл — см. tooling playbook (закрыть процесс / другой конфиг).
- **First adoption task:** минимальный цикл: правка XAML → build → ручная проверка сценария.
- **Success criterion:** нет накопления «красных» биндингов в логе при типичном сценарии.
- **Confidence:** high

---

### 12. Связь с кросс-стеком .NET UI

- **Fact:** паттерны MVVM и разделение состояния перекликаются с **WPF** и **MAUI**, но API контролов, стили и биндинги **не переносятся один в один**.
- **Heuristic:** при импорте знаний с WPF проверять **имена свойств, поведение layout, маршрутизацию команд**.
- **Доп. материал:** `frontend-dotnet-playbook.md` (Blazor / Avalonia / XAML обзор).
- **Confidence:** high
