## Синтез

Цель — собрать принципы и наблюдения для проектирования IDE и инструментов разработчика (в т.ч. Cascade IDE). Источники перечислены в Provenance.

### 1. Addy Osmani — The Developer Experience Book

#### Определения

- **DX** = опыт разработчика при использовании инструментов и сервисов для анализа, сборки, тестирования, интеграции, развёртывания и поддержки ПО. DX = UX для пользователей, которые являются разработчиками.
- **Ключевые черты хорошего DX:** usable, credible, findable, useful, valuable, accessible, desirable (адаптация семи факторов UX к разработчикам).

#### First Principles (принципы «с самого старта»)

1. **Create magical moments** — создавать мощные инструменты «неотличимые от магии», думать на 5–10 лет вперёд.
2. **Fight uncertainty** — не оставлять разработчика в неопределённости; быть opinionated, показывать «один верный путь» хотя бы на старте.
3. **Fight complexity** — решать за разработчика по умолчанию, но **финальное слово за ним**; усложнённое делать простым, но не делать простое сложным.
4. **Learning by doing** — обучение через действие: hands-on, эксперимент, guided flows, не только справочная документация.
5. **Incremental steps** — маленькие шаги, без «турбулентности»: информация и инструменты для перехода на следующий этап без перегрузки.
6. **Friction must be zero at T0** — нулевое трение в первые минуты: один клик — старт, возможность редактировать и запустить hello world без чтения доков.
7. **Fast is better than slow** — скорость и продуктивность в приоритете; минимум ожидания; метрики: время отклика, число кликов до фичи.
8. **Developer in control** — разработчик задаёт темп, имеет ресурсы и подсказки, но не обязан делать неожиданные действия; контроль и кастомизация.
9. **Show code in context** — показывать код в контексте реального приложения, не только «hello world» в изоляции.
10. **Decide for me, but let me have the final say** — сильные дефолты + escape hatches.

#### Что нужно разработчику (Tools & Languages)

- Быстрый поиск по коду (wildcard, идентификаторы, вызовы, использования).
- **IDE/LSP:** автодополнение, IntelliSense, **мгновенные сообщения об ошибках при наборе** (до рантайма/компиляции).
- Поддержка языков: линтинг, автоформатирование, подсветка, документация по месту вызова, jump to definition, запуск теста из редактора, отладка внутри редактора, плагины.
- Профилировщики CPU/памяти, хорошо интегрированные.
- Changelog по версиям; зависимости без конфликтов, подтягивание при отсутствии.

#### Ошибки и обратная связь

- **Хорошее сообщение об ошибке:** что случилось, причина, как исправить (и ссылка на док/решение).
- Обратная связь по мере набора (type checking, linting) — до появления ошибок компиляции/рантайма.
- Эмпатия, понятные формулировки, «путь к решению».

#### Документация

- Код в начале, не «зарывать lede»; несколько форматов (текст, видео, туториалы).
- Не только happy path: обходные пути, хаки, признание пробелов в продукте.
- Оптимизация под беглый просмотр: глаза идут к блокам кода; комментарии и несколько вариантов сниппетов.

#### Производительность и поток

- **Итерационная задержка** (время до отклика сборки/теста) критично: долгое ожидание → отвлечение, переключение контекста → потеря фокуса.
- Инструменты, которые трудно понять, тоже вызывают переключение контекста и нежелание менять код.
- Цель: разработчик тратит **>90% времени на программирование**, <10% на накладные расходы.

#### DX и AI (из книги)

- **Trust:** надёжность, предсказуемость; высокий precision на старте; объяснение релевантности; избегать непрошенных крупных переписываний; контекстная релевантность.
- **Familiarity:** опора на ментальные модели сотрудничества; со-обучение пользователя и ИИ; прозрачность данных и обратной связи; настраиваемость под workflow.
- **Control:** партнёрство «на равных»; opt-in/opt-out по фичам; возможность принять/проигнорировать/изменить/откатить предложения ИИ.

#### Измерение DX

- Usability: время на «getting started», время на выполнение задачи, размер бинарников, время сборки.
- Credibility: количество багов, совместимость, безопасность.
- Findability: время на поиск информации в продукте.
- Journey maps, friction audits, критические пути (toothbrush + pivotal journeys).

#### Цитаты для IDE

- *«Minimize switching contexts: interruptions matter even when it's your tools.»* (Kathy Korevec)
- *«Optimize for TTC (time to code): developers need to see code running to test, get feedback, and ship.»*
- *«You are a chef cooking for chefs — the bar is higher.»*

### 2. Smalltalk-80: The Interactive Programming Environment (Goldberg, 1984)

#### Структура книги (и среды)

1. **Введение в пользовательский интерфейс** — единый интерфейс системы.
2. **Поиск информации об объектах** — как находить сведения о объектах в системе.
3. **Исследование описаний классов и редакторы** — как исследовать классы и использовать редакторы для реализации новых классов.
4. **Поиск и исправление ошибок** — поддержка отладки.
5. **Доступ к внешним файлам и обслуживание системы** — восстановление после сбоев, управление изменениями.

#### Ключевые идеи для IDE

- **Единый мультицелевой интерфейс:** текст, графика, разработка программ, хранение и поиск информации — в одной среде.
- **Интегрированные инструменты:** компилятор, отладчик, текстовый редактор — в виде прямоугольных окон на битмап-дисплее; клавиатура + указательное устройство.
- **Немедленная компиляция и проверка** — без переключения режимов; пользователь остаётся в одном контексте.
- **Прозрачность:** «что видишь — то и имеешь»; объекты и их описание доступны для просмотра и изменения.
- Идеи Smalltalk-80 лежат в основе многих современных IDE: окна, инспекторы, браузеры классов, интеграция редактирования и выполнения.

#### Для идеального IDE

- Минимизировать переключение режимов (редактор / запуск / отладка) — всё в одном пространстве.
- Объекты/артефакты (файлы, классы, тесты) — с возможностью просмотра и навигации «из коробки».
- Восстановление после сбоев и управление изменениями — часть среды, а не отдельный инструмент.

### 3. diSessa — A Principled Design for an Integrated Computational Environment (Boxer, 1982)

#### Цель

Единая вычислительная среда для наивных и начинающих пользователей: редактирование текста, использование и изменение программ, написание программ, работа с данными, графика — через **общую вычислительную схему**, а не набор разрозненных подсистем.

#### Принципы понятности (understandability)

- **Структура vs функция:** структура — то, что определяет объект независимо от использования; функция — использование, цель, последствия. Плохая интеграция — когда у каждой «области» свои структуры и синтаксис; пользователь вынужден учить много похожего, но разного.
- **Два эвристических принципа интеграции:**
  1. **Строить функции из общих структур:** базовые структуры среды таковы, что конкретные функции (редактирование, выполнение, отладка) получаются из них. Тогда освоение одной области переносится в другие.
  2. **Shallow structuring:** любая важная для пользователя функция должна быть **простым сочетанием базовых структур**. Если разрыв между базовыми структурами и нужной функцией велик, для новичка он «бесконечен».
- **Единая схема вместо «структура на каждую функцию»:** избегать hacker bug (отдельная структура под каждую функцию) и formalist bug (слишком мало примитивов, всё через «трюки»).

#### Ментальные модели

- **Surrogate model** — целостная «заменительная» модель системы для предсказания и отладки; часто сложна и требует обучения; хороша для отладки, слаба для быстрого повседневного взаимодействия.
- **Functional model** — понимание через «что я хочу сделать» и «какое действие к этому ведёт»; контекстно, инкрементально учимо; но не даёт полной картины системы.
- **Distributed model** — множество частичных объяснений из разных контекстов (как «знать друга»); жизненно для сложных систем; опора на естественные рамки (например, язык, аналогии) и согласованность между контекстами.
- Для IDE: поддерживать и «быстрый путь» (функциональные сценарии), и возможность углубления (суррогатная модель через инспекторы, стек, данные).

#### Boxer и пространственная метафора

- **«What you see is what you have»** — наивный реализм: интерфейс не скрывает суть системы.
- **Пространственная метафора:** структуры и отношения в языке выражены пространственными отношениями на экране; опора на уже имеющиеся у человека пространственные схемы.
- Для IDE: визуальная структура (дерево решения, панели, связи файл–тест–диагностика) как часть модели понимания, а не только «красивый UI».

#### Синергия областей

- В интегрированной среде изучение одной функции автоматически даёт компетенцию в других (общие структуры, один синтаксис команд, единый стиль обратной связи).
- Редактор, выполнение, отладка, данные — не отдельные «приложения», а одна среда с разными точками входа.

### 4. Свод: принципы для идеального DX/IDE

| Область | Принцип | Источник |
|--------|---------|----------|
| Старт | Нулевое трение в первые минуты; один клик до «код запущен» | Osmani |
| Скорость | Быстро лучше медленно; минимум итерационной задержки; >90% времени на код | Osmani |
| Контекст | Минимизировать переключение контекстов; инструмент не должен прерывать поток | Osmani, Kathy Korevec |
| Ошибки | Сообщение: что случилось, почему, как исправить; обратная связь по мере набора | Osmani |
| Дефолты | Opinionated defaults + возможность кастомизации; «решай за меня, но последнее слово за мной» | Osmani |
| Обучение | Learning by doing; инкрементальные шаги; код в контексте, не только hello world | Osmani |
| Неопределённость | Бороться с неопределённостью; не оставлять разработчика «в подвешенном состоянии» | Osmani |
| Единая среда | Редактор, выполнение, отладка, навигация — одно пространство, без смены режимов | Goldberg, diSessa |
| Общие структуры | Одна схема/структура для разных функций; синергия при обучении | diSessa |
| Shallow structuring | Важная функция = простое сочетание базовых элементов; не «бесконечный» зазор до примитивов | diSessa |
| Прозрачность | «What you see is what you have»; визуальная структура отражает модель системы | Goldberg, diSessa |
| Контроль | Разработчик (и агент) в контроле; opt-in, настройка, откат | Osmani (DX + AI) |
| Документация | Найти за минуты; код и примеры в контексте; не только happy path | Osmani |

### 5. Связь с Cascade IDE и агентами

- **Единое окно, панели, состояние редактора** — в духе Smalltalk и Boxer: один контекст, минимум переключений.
- **MCP, Roslyn, dotnet-debug** — дают семантику и отладку «здесь же», без парсинга логов; быстрая обратная связь, точные координаты ошибок (file:line:column).
- **Agent-notes, Thought Stream** — вынос «рабочей памяти» вовне; борьба с «золотой рыбкой» (суммаризация) и когнитивной нагрузкой.
- **«Спроси модель», «как тебе?»** — снижение неопределённости и контекстное обучение (learning by doing в партнёрстве).
- **Идеальный DX/IDE для агента:** те же принципы (ноль трения, быстрый отклик, контекст, контроль) + явная модель среды (что видит агент = что есть), общие структуры (решение, файлы, диагностики, тесты) и возможность дополнять среду (заметки, тулы).
