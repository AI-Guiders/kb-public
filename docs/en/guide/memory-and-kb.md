# Memory and KB: how it works

**Three contours** (personal canon · group KB · kb-public · workspace): [Three contours map](../onboarding/three-contours.md).  
**Layers L0–L3 and publishing:** [kb-one-pager-structure-and-protocols-v1.md](../knowledge/kb-one-pager-structure-and-protocols-v1.md)

## Why it exists

KB keeps the environment **stable across switches, pauses, and context changes**.

## Memory model (practical)

- **Hot context (L0/L1)** — what you need right now for the task.
- **Working rules/protocols** — stable agreements (decisions, mode switches, validation).
- **Archive/history** — past decisions and traces needed less often.

## What to write

Only what:

- will be reused;
- reduces repeated explanations;
- prevents typical mistakes.

Do not write “daily noise” that helps nobody tomorrow.

## Good record format

A good note answers:

1. Context: when it applies.
2. Decision: what we do.
3. Boundaries: where it does **not** apply.
4. Verification: how to tell it was applied correctly.

## Quick context recovery

If you fell out of flow:

1. Open [home](../index.md).
2. Read [Entry KB](../knowledge/00-entry-kb-v1.md).
3. Restore current goal and last recorded step.
4. Continue with the nearest verifiable action.

## Healthy KB sign

A new participant can enter the work loop without a two-hour call.
